A01 – Broken Access Control

Lỗ hổng phổ biến nhất trên web: user truy cập được resource hoặc chức năng mà họ không được phép. Số 1 liên tiếp vì cực khó detect tự động và dễ bị bỏ sót khi review code.

100%
app có ít nhất 1 BAC issue (theo OWASP 2025)
40
CWE được map vào category này

Concept

Broken Access Control là gì?

🔑 Một câu để nhớ
Access control gồm 3 lớp cần phân biệt rõ: Authentication (bạn là ai?), Session management (request này có phải của bạn không?), và Access control (bạn có quyền làm điều này không?). BAC xảy ra khi lớp thứ 3 bị bypass — app biết bạn là ai nhưng không kiểm tra bạn được làm gì.

Access control enforcement cần xảy ra ở server-side. Nếu chỉ kiểm tra ở client (JavaScript, hidden field, URL param), attacker bypass dễ dàng bằng Burp Suite hoặc curl.

Attack types

Các dạng tấn công chính

IDOR
Critical
›
Insecure Direct Object Reference – đổi ID để xem data của người khác.

App dùng user-controlled input để truy xuất object trực tiếp mà không verify ownership.

# Thử đổi id từ 123 → 124
GET /api/orders/123    → 200 OK (đơn của bạn)
GET /api/orders/124    → 200 OK (đơn của người khác!)

GUID không bảo vệ được nếu GUID bị leak trong response API khác (e.g. user reviews, messages).

Vertical Privilege Escalation
Critical
›
User thường truy cập được chức năng chỉ dành cho admin.

App không bảo vệ admin endpoint, hoặc kiểm tra role ở client-side.

# Force browse đến admin page
GET /admin/deleteUser  → 200 OK (không check role!)

# Parameter tampering
?role=admin  hoặc  Cookie: role=admin
Horizontal Privilege Escalation
High
›
Truy cập resource của user cùng cấp quyền nhưng khác account.

Dạng IDOR nhưng target là user cùng role. Đặc biệt nguy hiểm: nếu target là admin thì leo thang thành vertical escalation.

GET /myaccount?id=456
# Nếu id=456 là admin → lấy được admin creds
Force Browsing
Medium
›
Đoán/bruteforce URL để truy cập page không có link trên UI.

App ẩn URL thay vì enforce authorization — "security by obscurity" không phải access control thật.

# Trong robots.txt hoặc JS source code
/admin
/dashboard/internal
/api/v1/admin/users
/administrator-panel-yb556

# Script leak trong HTML
if (isAdmin) { href = '/admin-panel' }
JWT / Metadata Tampering
High
›
Sửa token, cookie, hidden field để nâng quyền.

App lưu role trong JWT/cookie mà user có thể sửa.

# JWT payload decode
{ "user": "alice", "role": "user" }
# Sửa thành
{ "user": "alice", "role": "admin" }
Platform Misconfiguration
Medium
›
Bypass rule bằng non-standard header hoặc HTTP method.

Rule block POST nhưng không block GET. Hoặc dùng header để override URL.

# Header bypass
POST / HTTP/1.1
X-Original-URL: /admin/deleteUser

# Method bypass (thay POST → GET)
GET /admin/deleteUser?userId=5

# Trailing slash bypass
GET /admin/deleteUser/
⚠️ Hai dạng hay bị bỏ sót
  • CORS misconfiguration — cho phép origin không tin cậy gọi API authenticated. Nguy hiểm khi kết hợp với XSS.
  • Multi-step process bypass — step 1 và 2 check quyền, nhưng step 3 (confirmation) không check. Attacker skip thẳng đến step 3.
Detection

Dấu hiệu nhận biết

  • !
    URL chứa numeric ID
    /users/123, /orders/456, /api/files/789 — luôn thử đổi sang ID khác
  • !
    Parameter chứa role/admin/privilege
    ?admin=false, ?role=user trong URL hoặc body — test thay đổi giá trị
  • ~
    JWT/Cookie chứa role claim
    Decode JWT payload → tìm "role", "isAdmin", "permissions"
  • ~
    Admin URL lộ trong source code hoặc JS
    Inspect HTML/JS tìm link ẩn, check /robots.txt, sitemap.xml
  • i
    Response trả về HTTP redirect nhưng vẫn có data
    App trả 302 Found nhưng body response vẫn chứa sensitive data — Burp bắt được response đầy đủ trước khi browser redirect
  • i
    API endpoint POST/PUT/DELETE không có auth check
    Frontend có kiểm tra nhưng API raw không — test trực tiếp bằng Burp hoặc curl
Attack flow

Methodology – Tư duy attacker

🎯 IDOR Testing
1
Identify endpoints có ID/parameter
Scan toàn bộ request trong Burp, tìm numeric/predictable ID trong URL, body, header
2
Tạo 2 account (A và B)
Dùng account A để tạo resource, ghi lại ID. Dùng account B để request ID đó.
3
Thay đổi ID trong Burp Repeater
Test thủ công, dùng Intruder để enumerate, dùng Autorize để automate cross-account
4
Observe response
200 + data = IDOR confirmed. 403 = có check. 200 nhưng empty có thể vẫn là bug (enumeration).
5
Escalate: test write operations
PUT/DELETE/POST với ID của người khác — impact cao hơn nhiều so với chỉ read
🎯 Privilege Escalation Testing
1
Map chức năng admin (với account admin nếu có)
Dùng Burp để capture tất cả request khi dùng admin account, ghi lại endpoint
2
Force browse các admin URL bằng user session
Dùng Autorize: load admin sitemap, replay tất cả request với user token
3
Test parameter/cookie manipulation
Tìm và sửa role-related value, test cả unauthenticated (bỏ cookie hoàn toàn)
Payloads

Payload & Request mẫu

# Horizontal IDOR – đổi ID
GET /api/users/1337/profile HTTP/1.1
Authorization: Bearer <your_token>

# IDOR qua body parameter
POST /api/transfer HTTP/1.1
{
  "from_account": 99999,
  "to_account": "attacker",
  "amount": 1000
}

# IDOR với static file
GET /files/../user_456/contract.pdf
# Force browse admin page
GET /admin HTTP/1.1
GET /admin/users HTTP/1.1
GET /management/console HTTP/1.1

# Parameter tampering trong URL
GET /home?admin=true HTTP/1.1
GET /login?role=administrator HTTP/1.1

# Cookie manipulation
Cookie: session=abc123; role=admin

# Hidden field (capture từ form → sửa)
<input type="hidden" name="role" value="admin">
# Header override bypass
POST / HTTP/1.1
X-Original-URL: /admin/deleteUser
X-Rewrite-URL: /admin/deleteUser

# HTTP method change (POST → GET)
GET /admin/deleteUser?userId=5 HTTP/1.1

# URL case variation
GET /ADMIN/deleteUser HTTP/1.1
GET /Admin/DeleteUser HTTP/1.1

# Trailing slash bypass (Spring useSuffixPatternMatch)
GET /admin/deleteUser/ HTTP/1.1
GET /admin/deleteUser.json HTTP/1.1

# Referer spoofing (nếu app dùng Referer để auth sub-pages)
GET /admin/settings HTTP/1.1
Referer: https://target.com/admin

echo "eyJ..." | base64 -d

# 2. Payload thường gặp
{
  "sub": "user123",
  "role": "user",      ← thử sửa thành "admin"
  "iat": 1700000000,
  "exp": 1700086400

# Bỏ signature, server không verify

Prevention

Cách fix – Cho developer và non-tech

💡 Nguyên tắc cốt lõi: "Deny by default"
Thay vì liệt kê những gì bị cấm, hãy chỉ cho phép những gì được phép rõ ràng. Mọi resource mặc định là không ai được truy cập, trừ khi có rule cụ thể grant quyền.
Server-side enforcement
Check quyền ở server — không trust input từ client (URL param, cookie, hidden field, header).
Ownership validation
Với mỗi resource, kiểm tra user hiện tại có phải owner không — không chỉ check authen.
RBAC (Role-Based Access Control)
Định nghĩa rõ role và permission. Implement tập trung một lần, reuse toàn app.
Rate limiting API
Giới hạn số request để ngăn automated enumeration (brute-force ID, force-browsing).
Short-lived JWT
JWT expired nhanh (15–60 phút). Dùng refresh token. Invalidate token sau logout.
Log & Alert
Log access control failures. Alert admin khi có repeated failures — dấu hiệu của enumeration attack.

❌ Bad practice vs ✅ Good practice

❌ Frontend-only check
if (user.role === 'admin') { showAdminButton(); } // API không check gì cả
✅ Server-side check
// Mỗi API endpoint if (!isAuthorized(user, 'admin')) { return 403 Forbidden; }
❌ Không check ownership
GET /orders/{id} // Trả về luôn, không check // order có thuộc user không
✅ Verify ownership
order = db.getOrder(id) if (order.userId != currentUser.id): return 403 Forbidden

Autorize (Burp extension)

Real-world

Case thực tế điển hình

Case 1 – IDOR lộ đơn hàng khách hàng khác (E-commerce)
App thương mại điện tử để order_id là số tăng dần trong URL: /orders/10234. Attacker thay đổi ID tuần tự → xem được tên, địa chỉ, sản phẩm đơn hàng của toàn bộ khách hàng. Root cause: API GET /orders/{id} không kiểm tra order đó có thuộc về user đang login không.
Case 2 – Admin panel không có auth check (Real bug)
Ứng dụng nội bộ có /admin/console được access control bằng JavaScript (ẩn button). Attacker dùng curl https://target.com/admin/console → response đầy đủ HTML admin panel. Không cần bypass gì thêm vì server không check session.
Case 3 – Horizontal → Vertical escalation qua password reset
Trang /myaccount?id=456 của admin hiển thị form đổi password. Attacker (đang login với user thường) đổi id sang ID của admin → thấy form → submit password mới → login admin account → vertical privilege escalation thành công.
Mindset

Attacker mindset – Cách suy nghĩ khi test

"Mỗi khi thấy ID trong request, câu hỏi đầu tiên: server có verify object này thuộc về mình không?"
Không phải đổi ID là luôn ra IDOR — nhưng luôn phải thử. Ngay cả GUID cũng có thể bị leak từ endpoint khác.
"Mỗi khi thấy role/admin trong request, câu hỏi: ai control giá trị này?"
Nếu giá trị đến từ client (cookie, URL param, JWT payload không verify), đó là attack surface. Server phải tự lấy role từ DB, không được tin client-sent role.
"Frontend không hiển thị button/link không có nghĩa là endpoint được bảo vệ."
Security by obscurity ≠ access control. Luôn test API endpoint trực tiếp, bỏ qua UI. curl và Burp không bị ảnh hưởng bởi JavaScript UI logic.
